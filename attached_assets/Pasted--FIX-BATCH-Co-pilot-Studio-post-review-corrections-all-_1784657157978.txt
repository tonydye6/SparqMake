════════ PHASE A — FLOW-CRITICAL (the app is visibly broken without these) ════════

A1. Session view is blind: variants always parse to []
- File: artifacts/sparqmake/src/pages/CopilotStudio.tsx ~line 546 (loadSession)
- Bug: GET /api/creatives/:id/variants returns a BARE ARRAY, but the code reads
  `vData.variants || vData.data || []`, so allVariants is always empty: the main preview,
  history-rail thumbnails, and turn thumbnails never show images.
- Fix: parse defensively: `const variants = Array.isArray(vData) ? vData : (vData.variants
  || vData.data || [])`. Keep everything downstream unchanged.
- Verify: open an existing session with prior turns: preview shows the active image, history
  tiles show thumbnails, compare grids show take images.

A2. Reference images + video seed never reach the model (nonexistent `media` field)
- File: artifacts/api-server/src/services/interactions-client.ts (runImageInteraction ~136-153,
  runVideoInteraction ~72-101)
- Bug: reference slots and the video seed are sent as a top-level `media: [...]` on the
  interactions.create request. CreateModelInteraction has NO media field (the `as
  Parameters<...>[0]` cast hides this); the API reads model input ONLY from `input`. Result:
  brand/persona reference conditioning and image-to-video seeding are silent no-ops.
- Fix: build `input` as a Content array per @google/genai types: text part(s) plus one
  image part per reference: `input: [{ type: "text", text: prompt }, { type: "image",
  data: <base64>, mime_type: <mime> }, ...]`. Same for the video seed image. Remove the
  `media` field and remove the blanket `as Parameters<...>[0]` cast (type the request
  properly so future drift is caught by tsc).
- Verify: (1) unit test asserting the built request has no `media` key and that input
  contains N image parts; (2) live check: generate a draft with a distinctive subject
  reference (e.g. the Crown U mascot asset) and confirm the output visibly contains the
  referenced subject; convert an image to video and confirm the clip is derived from that
  image (not an unrelated scene).

  3) Fix idx-22's `when` in _journal.json to a value between 0021's and 0023's timestamps so

  4) Apply migrations in dev and confirm both fresh-DB and already-applied-DB paths succeed.
- Verify: `drizzle-kit generate` runs clean afterward with NO new diff; migrate exits 0 on a

A4. Composer keyword regex hijacks image edits
- File: artifacts/sparqmake/src/pages/CopilotStudio.tsx ~line 668 (handleSend)

A5. Branching from non-image turns kills the edit chain

════════ PHASE B — MONEY / BUDGET INTEGRITY ════════

B1. Unify the daily-budget reservation (lock-key race + duplicated logic)
- Files: artifacts/api-server/src/routes/sessions.ts ~174-220; generate.ts (reserveBudget
  ~1131-1195); video.ts ~52
- Bug: sessions.ts re-implements reservation with BUDGET_LOCK_KEY=100002 while generate.ts and
  video.ts use 100001 over the same cost_logs SUM: the paths do not mutually serialize, so a
  concurrent turn + /generate can jointly exceed dailyCostThreshold. The sessions 429 body also
  omits the `message` field the shared helper returns.
- Fix: export reserveBudget()/budgetExceededBody() from a shared module (e.g.
  lib/budget.ts inside api-server) with the SINGLE lock key 100001; delete the copy in
  sessions.ts and call the shared helper everywhere (generate.ts, video.ts, sessions.ts).
- Verify: grep shows exactly one pg_advisory_xact_lock call site/constant; sessions 429
  response includes `message`.

B2. Reservation settlement must be awaited and atomic
- File: sessions.ts ~264-279
- Bug: the reservation row delete is fire-and-forget in `finally`
  (`db.delete(...).catch(log)`, never awaited) while the real cost row is written earlier
  inside executeTurn: a failed delete or a restart leaves a phantom reservation that the
  day-spend SUM counts → wrongful 429s for the rest of the day.
- Fix: settle like generate.ts does: pass the reservationId into the turn pipeline and, in the
  SAME transaction that writes the real cost_logs row (session-service executeTurn ~467),
  delete the reservation. On the error path, await the delete before rethrowing.
- Verify: kill the process mid-turn in dev: no budget_reservation row remains after the
  transaction boundary test; unit test covers reserve→settle and reserve→error→release.

B3. Reserve what turns can actually spend (QA + outpaint allowances)
- Files: sessions.ts estimateTurnCost ~394-415; session-service.ts applyQaPass ~1034-1080,
  executeFanOut ~1481-1650
- Bug: the QA corrective pass spends an extra estimateImagenCost(1) and fan-out spends
  estimateImagenCost(perOutpaintedPlatform), neither reserved: real spend slips past the gate.
- Fix: draft/edit_image/edit_region reservations include + estimateImagenCost(1) QA allowance;
  fan_out reservation includes estimateClaudeCost() + estimateImagenCost(selectedPlatformCount)
  as worst-case outpaint allowance. Also delete the duplicated per-action cost formulas:
  export a single estimateActionCost(action, params) from session-service and use it for BOTH
  the reservation and the charged costUsd so they cannot drift.
- Verify: unit test asserts reservation >= charged cost for draft-with-QA and a fan-out where
  every platform outpaints.

B4. One video pricing model
- Files: ai-config.ts ~24-35; session-service.ts ~1324-1325, ~1403-1404; video.ts ~208-215;
  video-generation.ts estimateVideoCost ~113-115
- Bug: three inconsistent formulas: flat 2.10 reservation, byte-heuristic 0.42/s charge, and
  legacy Veo 0.35/s×6s still logged for Omni output in video.ts.
- Fix: single helper: estimated duration from the requested duration (see D2: we now request
  a fixed duration) with the byte heuristic only as fallback; price = duration ×
  VIDEO_COST_PER_SECOND_USD; use it for reservation AND charge in both the copilot and legacy
  video paths; delete estimateVideoCost's stale 0.35 constant.
- Verify: cost_logs rows for a copilot video and a legacy video show the same formula; unit
  test pins reservation == charge for a fixed-duration clip.

════════ PHASE D — INTERACTIONS CLIENT & VIDEO HARDENING ════════

D1. Timeouts + cancellation end-to-end
- Files: interactions-client.ts ~101, ~170; sessions.ts ~224-279; video-generation.ts ~59-79;
  CopilotStudio.tsx runTurn ~604-650
- Bug: no timeout/AbortSignal on any interactions.create (SDK supports
  options.timeout and fetchOptions.signal); the route's SSE timeout/close never aborts
  executeTurn; the FE never aborts its fetch on unmount. Hung calls = turns stuck 'running',
  phantom reservations, and abandoned turns that keep billing.
- Fix: (a) interactions-client accepts an optional AbortSignal + applies timeouts (image 180s,
  video 300s) via the SDK options; (b) sessions.ts creates an AbortController, aborts it on
  req 'close' AND on the SSE timeout, and threads the signal through executeTurn into every
  model call (mark the turn 'error'/aborted on abort, settle the reservation per B2);
  (c) video-generation.ts passes the signal via fetchOptions.signal (restoring the
  cancellation the Veo path had); (d) the FE creates an AbortController per runTurn, aborts on
  unmount/back, and ignores dispatches after abort.
- Verify: start a draft, navigate Back immediately: server logs show the turn aborted, turn
  status='error', no reservation row remains; a mocked never-resolving model call times out at
  the configured bound.

D3. Robust SSE parsing + inline delivery + input validation edges
- Files: CopilotStudio.tsx ~603-643; interactions-client.ts ~73-75, ~105/174, ~146;
  sessions.ts CreateTurnBody ~78; session-service.ts ~1285-1287
- Bug: (a) FE SSE parser pairs `event: error` with its data line via lines.indexOf()+1 and an

  extraction reads only output_image/output_video `.data` (no delivery requested, no `.uri`

  a whitelist (nearest of 16:9 / 9:16 / 1:1) before response_format.
- Verify: unit tests: chunk-split error event surfaces as a thrown turn error; platform:""
  is rejected 400; linkedin-sourced video request carries 16:9.
